# 12장 창발성

## 창발적 설계로 깔끔한 코드를 구현하자

켄트 벡이 제시한 단순한 설계 규칙 네가지가 소프트웨어 설계 품질을 크게 높여준다

- 모든 테스트를 실행한다.
- 중복을 없앤다.
- 프로그래머 의도를 표현한다.
- 클래스와 메서드 수를 최소로 줄인다.

→ 목록은 중요도 순으로 나열

## 모든 테스트를 실행하라

설계는 의도한 대로 돌아가는 시트템을 내놓아야 한다.

→ 테스트가 불가능한 시스템은 검증도 불가능!

테스트가 가능한 시스템을 만들 떄 설계 품질이 더불어 높아진다.

- SRP를 준수하는 클래스는 테스트가 더 쉽다.
- 결합도가 높으면 테스트가 더 어렵다.

→ 테스트 케이스를 작성하면 설계 품질이 높아진다.
(시스템은 낮은 결합도와 높은 응집력이라는, 객체 지향 방법론이 지향하는 목표 저절로 달성)

## 리팩터링

테스트 케이스 작성 후 → 코드와 클래스 점진적으로 리팩터링

- 코드를 정리하면서 시스템 깨질까 걱정 필요 X ⇒ 테스트 케이스 존재!

### 중복을 없애라

우수한 설계에서 중복은 커다란 적!

→ 단 몇 줄이라도 중복을 제거하겠다는 의지 필요

### 표현하라

자신이 이해하는 코드를 짜기는 쉽다 
→ BUT 유지보수 개발자가 시스템을 이해하느라 보내는 시간은 늘어나고 코드 오해 가능성 커진다.

표현력을 높이는 방법

- 좋은 이름을 선택한다.
- 함수와 클래스 크기를 가능한 줄인다.
- 표준 명칭을 사용한다.
- 단위 테스트 케이스를 꼼꼼히 작성한다.(테스트 케이스 = 예제로 보여주는 문서)

→ 표현력을 높이는 방법 중 가장 중요한 것은 노력이다.

### 클래스와 메서드 수를 최소로 줄여라

목표 : 클래스 크기를 작게 유지하면서 동시에 시스템 크기도 작게 유지하는 것

클래스와 함수 수를 줄이는 작업 중요

→ BUT 테스트 케이스 생성하고 중복을 제거하고 의도를 표현하는 작업이 더 중요

## 결론

경험을 대신할 단순한 개발 기법은 당연히 없다!

→ 단순한 설계 규칙을 따른다면 우수한 기법과 원칙을 단번에 활용할 수 있다
